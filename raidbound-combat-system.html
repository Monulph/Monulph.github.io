Raidbound Combat System

Deterministic, party-based encounters built around clarity, pacing, and strong UI feedback

A tactical roguelite combat layer designed for readability first

← Back to Home

Raidbound's combat is built to feel strategic without becoming slow or messy. My goal is to combine the readable menu flow of classic RPG combat with a faster, more modern pace driven by ATB charges, threat generation, and strong feedback.

  • Threat-based encounters
  • ATB-driven decisions
  • Menu clarity first
  • Deterministic pacing
  • Readable feedback

Core Goals

Readable at a glance

Heroes, enemies, ATB readiness, damage events, and targeting state need to be understandable quickly. UI readability is part of the combat design, not decoration added later.

Fast but controlled

Encounters move faster than a traditional turn-based battle, but actions still feel deliberate. The player is making tactical choices instead of waiting through long dead time.

Party synergy matters

Heroes are designed around roles, buffs, debuffs, healing, threat generation, and item interactions so runs feel like party-building problems instead of isolated button presses.

How a Fight Works

1. Encounter begins

Combat loads from the map flow, presents enemy composition, and sets the player into a clean battlefield view with immediate information about the incoming pack.

2. ATB fills

Heroes and enemies build toward actions over time. When a hero reaches a full action threshold, the player can choose to attack or use an ability.

3. Threat and effects resolve

Healing, damage, taunts, buffs, debuffs, and item effects resolve through shared combat rules so hero kits stay flexible without becoming one-off scripts.

4. Feedback sells the hit

Card shake, floating numbers, glow states, and transitions reinforce the result of each action so the battle feels alive even though it is menu driven.

Why I Built It This Way

I wanted Raidbound to avoid two common problems: combat that feels cluttered and combat that feels passive. The structure is intentionally deterministic so I can balance it more reliably, build better tools around it, and keep the experience legible even as more heroes, items, and enemy mechanics are added.

The combat system is also being shaped around maintainability. Refactors toward schema-first data, shared ability logic, and cleaner screen responsibilities make it easier to expand the game without dragging legacy behavior forward.

What This System Supports

Hero identity

Distinct classes and roles with their own strengths, ability patterns, and synergy hooks.

Item-driven builds

Combat stats and effects can shift over a run, creating different solutions to the same encounters.

Encounter variety

Normal, elite, rare, and boss fights can change pacing and decision pressure without breaking the base rules.

Future web demo slicing

The screen-based flow and deterministic structure make combat a strong candidate for a focused browser-playable slice.
