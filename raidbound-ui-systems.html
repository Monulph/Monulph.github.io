Raidbound is a UI-heavy game. Because the player is reading enemy states, hero readiness, loot, map progression, and collection screens constantly, interface quality directly affects whether the game feels strategic or overwhelming.

UI Priorities

Readability first

Critical information should be easy to parse under pressure. That includes combat readiness, reward choices, and party information.

Consistent framing

Panels, borders, spacing, and button treatment should feel like part of one visual language across menus, overlays, and reward screens.

Dark-fantasy presentation

The game aims for a moody, painterly identity while still keeping contrast high enough that interaction remains comfortable.

Screens I Focus On Most

Combat cards

Enemy and hero cards must communicate state, portrait identity, timing, and impact feedback without feeling noisy.

Map flow

The map should feel cinematic while still supporting player navigation, node understanding, and run pacing.

Reward screens

Loot and collection screens need enough structure that the player can compare options quickly and feel rewarded, not interrupted.

Collections and options

Longer-term utility screens need to stay clean even as the project gains more heroes, items, and settings.

Why This Page Matters in My Portfolio

A lot of portfolio work shows art or code in isolation. UI systems show how those pieces meet. This page reflects the kind of work that sits between design, implementation, playtesting, and polish: layout decisions, consistency passes, feedback tuning, navigation flow, and reduction of clutter.

For Raidbound, that work is not secondary. It is one of the reasons the project feels more complete every time a pass lands.
